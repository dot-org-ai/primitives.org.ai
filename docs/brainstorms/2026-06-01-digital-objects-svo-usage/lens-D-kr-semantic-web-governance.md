# Lens D — Knowledge Representation / Semantic Web / Canonical-Vocabulary Governance

**Date:** 2026-06-01 · **Mode:** HOW (generative) · **Unit of analysis:** the vocabulary as a shared, portable, governed, evolving artifact across many domains and orgs.

## Lens thesis

The `digital-objects` SVO layer is not just a runtime for one app — it is a **vocabulary factory that mints linked data.** Every `Ontology()` a startup defines is simultaneously a private domain model AND a publishable fragment of a global knowledge graph. The three load-bearing fields — `Verb.source` (provenance), `Verb.canonical` (registry status), and graphdl's `$type` type-URI — turn each local schema into something that can be *grounded* against canonical registries (`verbs.org.ai`, `apqc`, `onet`, `schema.org.ai`, FIBO, NAICS), *exported* as a JSON-LD `@context`/`@graph`, *versioned* like any ontology, and *fused* with other orgs' graphs because content-derived IDs give the same Thing the same global address. When `explore.startups.studio` generates a million startups, the win is not a million islands — it is **one navigable, queryable, composable web of grounded predicates**, where any two startups that both say `pays` against the same FIBO predicate are interoperable for free. This lens treats the SVO model the way the W3C treats RDF: a Verb is an `rdf:Property` with `rdfs:domain`/`rdfs:range`, a Noun is an `rdfs:Class` with a resolvable IRI, an inverse is `owl:inverseOf`, and the whole ontology is a citable, dereferenceable artifact at a URL.

## Use cases

### 1. Importing O*NET tasks as canonical Verbs into a `digital-tasks` ontology
**Use case.** A staffing-automation startup wants its `digital-tasks` model to speak the language of real occupations. It imports the O*NET task statements for `15-1252.00` (Software Developers) — "Analyze user needs and software requirements", "Modify existing software to correct errors" — minting Verbs `analyzeRequirements`, `modifySoftware` stamped `source: 'onet'`, `canonical: true`, each carrying the O*NET task IRI (`onet:task/15-1252.00/...`) as its grounding. The Frame is derived from the task statement's complement structure (`{ subject: 'Developer', object: 'Software', topic: 'Requirement' }`).
**KR/standards mechanism.** O*NET's occupation→task taxonomy (SOC code as occupation IRI, numbered task statements as predicate sources) imported as canonical Verbs with external provenance.
**DEMAND.** `Verb.source: 'onet'` + `canonical: true` must round-trip through a registry import, and a Verb must carry a *resolvable external IRI* for the registry entry it grounds against (not merely the enum tag `'onet'`). The interface needs a place to put `onet:15-1252.00#analyze` — i.e. a `groundedAs?: IRI` slot, or the convention that `Verb.name` resolves to one.

### 2. APQC process verbs feeding `business-as-code`
**Use case.** A back-office automation builder grounds its operational vocabulary in APQC's Process Classification Framework. "Develop business strategy" (PCF `1.2`, element `10015`) and "Procure materials and services" (PCF `4.2`) become Verbs `developStrategy`, `procureMaterials` stamped `source: 'apqc'`, each carrying APQC's *dual identifier* — the hierarchy number (`1.2`) and the stable 5-digit element ID (`10015`). When APQC renames the process across PCF versions, the local Verb stays bound by the stable numeric ID, not the label.
**KR/standards mechanism.** APQC PCF's dual-ID scheme (mutable hierarchy ID + immutable element ID) — exactly the SKOS pattern of `skos:notation` + `prefLabel`.
**DEMAND.** A Verb's grounding identity must be the **stable element ID**, decoupled from the display label, so a label rename in the upstream registry does not break the grounding. The interface needs the Verb's external anchor to be an opaque stable code, with `name`/`action`/`event` as the human-facing layer over it.

### 3. Stamping a domain verb, then promoting it to `verbs.org.ai`
**Use case.** A logistics startup defines `palletize` locally — no canonical registry has it. `defineVerb` defaults it to `source: 'domain'`, `canonical: false`. Across the million-startup graph, 4,000 other logistics startups independently mint `palletize`/`palletise`/`stackOntoPallet`. A governance crawler clusters these by embedding + Frame signature, observes the convergence, and **promotes** a canonical `palletize` into `verbs.org.ai` with a published IRI. The 4,000 local Verbs are then *re-bound*: `source: 'domain' → 'verbs.org.ai'`, `canonical: false → true`, with `canonicalOf` pointing at the new global IRI.
**KR/standards mechanism.** Bottom-up vocabulary emergence (the way `schema.org` actually grew — `Action`, `Recipe`, `JobPosting` were proposed from usage), with provenance promotion.
**DEMAND.** `source` and `canonical` must be **mutable post-mint** and the promotion must be a first-class state transition (`domain/false → verbs.org.ai/true`) that preserves the local Verb's identity while pointing it at a newly-canonical IRI. The interface needs a `canonicalOf?: IRI` back-link and the guarantee that promoting a Verb does not orphan existing Actions that reference it by name.

### 4. Nouns `$type`'d to schema.org / schema.org.ai classes
**Use case.** A CRM-builder declares `Ontology({ Contact: { $type: 'https://schema.org/Person', ... }, Company: { $type: 'https://schema.org/Organization' } })`. Because `Contact` is grounded to `schema:Person`, the startup's contact records auto-emit schema.org `Person` microdata/JSON-LD on its public pages, get picked up by search-engine rich results, and interoperate with any CRM that also grounds Contact to `schema:Person`.
**KR/standards mechanism.** schema.org `rdfs:Class` reuse via graphdl's `$type` type-URI (already supported: `Graph({ User: 'https://schema.org.ai/Person' })`).
**DEMAND.** Every Noun must resolve to a `$type` IRI (defaulting to `schema.org.ai/{Name}` when unspecified), and the `Ontology()` factory must thread graphdl's `$type` through to the runtime `Noun` so a `Thing` can serialize with `"@type": "Person"`. The committed `Noun` interface (which lacks a `$type` field) must gain one or inherit graphdl's.

### 5. Importing and extending a standard schema (subclassing)
**Use case.** A fintech startup imports FIBO's `LoanContract` and FIBO's `pays`/`owesTo` predicates, then *extends* `LoanContract` with a domain subclass `MicroLoan` (`rdfs:subClassOf fibo:LoanContract`) and a domain Verb `microRepay` declared as a sub-predicate (`rdfs:subPropertyOf fibo:pays`). A solvency query that asks "show me everything that `fibo:pays`" automatically includes `microRepay` Actions via the subproperty entailment.
**KR/standards mechanism.** FIBO (Financial Industry Business Ontology) class/predicate import + `rdfs:subClassOf`/`rdfs:subPropertyOf` extension.
**DEMAND.** Nouns need a `subClassOf?: IRI` and Verbs need a `subPropertyOf?: IRI` slot so the local vocabulary can declare itself a specialization of an imported standard — enabling subsumption queries that fold domain Verbs under their canonical parent.

### 6. Frame domain/range mapping to `rdfs:domain` / `rdfs:range`
**Use case.** A startup's Verb `hires` declares `Frame { subject: 'Company', object: 'Person' }`. On export, the Frame's `subject` becomes `rdfs:domain schema:Organization` and `object` becomes `rdfs:range schema:Person`. A reasoner ingesting the exported graph can now *infer* that any Thing appearing as the subject of a `hires` Action is a `Company`, even if its `$type` was never explicitly stated — classic RDFS domain entailment.
**KR/standards mechanism.** `rdfs:domain`/`rdfs:range` derived directly from `Frame.subject`/`Frame.object`; the other roles (`recipient`, `instrument`, `source`) map to additional argument-role properties (akin to schema.org `Action`'s `agent`/`object`/`instrument`/`recipient` — which is *literally already this frame*).
**DEMAND.** The `Frame` interface's role slots must export to a well-defined RDF predicate vocabulary. schema.org `Action` already defines `agent`/`participant`/`object`/`result`/`instrument`/`recipient` — the Frame roles must map onto those names so an exported Action is a valid schema.org `Action`. (`subject→agent`, `object→object`, `recipient→recipient`, `instrument→instrument`, `result→result`.)

### 7. A verb's inverse mapping to `owl:inverseOf`
**Use case.** `hires` declares `inverse: 'employedBy'`. On export, the ontology emits `verbs:hires owl:inverseOf verbs:employedBy`. Any consumer graph that asserts `Ada employedBy Acme` can, under OWL reasoning, materialize `Acme hires Ada` without a second edge — and the `reverse: { by, at, in, for }` forms (`hiredBy`, `hiredAt`, `hiredIn`) become the surface labels for traversing the inverse direction.
**KR/standards mechanism.** `owl:inverseOf` derived from graphdl's `Verb.inverse` + the nested `reverse` conjugation block.
**DEMAND.** `Verb.inverse` must export to `owl:inverseOf` and the `reverse.{by,at,in,for}` forms must be carried through to the runtime Verb so traversal-by-inverse and NL-rendering of the reverse direction both work off the same declaration. The DECISION to use graphdl's nested `reverse` (not flat `reverseBy/At/In`) is the right one for this — it keeps the inverse a structured object, not three loose strings.

### 8. Exporting an entire startup's ontology as a portable JSON-LD `@graph`
**Use case.** `explore.startups.studio` lets a founder click "Export ontology." The studio serializes the startup's full `Ontology()` — Nouns as `rdfs:Class` nodes, Verbs as `rdf:Property` nodes with `rdfs:domain`/`range`/`inverseOf`, all `$type` groundings — into a single JSON-LD document: a top-level `@context` mapping every CURIE prefix (`schema:`, `verbs:`, `apqc:`, `onet:`, `fibo:`) plus an `@graph` array of every class and predicate. The founder publishes it at `https://acme.studio/ontology.jsonld`; it is now dereferenceable, linkable, and importable by anyone.
**KR/standards mechanism.** JSON-LD `@context` + `@graph` serialization of the whole vocabulary as a single citable artifact.
**DEMAND.** The `Ontology()` object must **serialize losslessly to a JSON-LD `@context` + `@graph`** and re-import without information loss — every Frame role, every `source`/`canonical` flag, every `$type`, every inverse must survive the round-trip. This is the single biggest demand on the interface (see "highest-leverage").

### 9. Two orgs interoperating because both ground `pays` against the same FIBO predicate
**Use case.** A B2B marketplace (org A) and a billing SaaS (org B) never coordinated. Both grounded their payment Verb to `fibo:pays`. When org B ingests org A's exported `@graph`, it does NOT need a mapping layer: A's `settle` and B's `remit` are both `rdfs:subPropertyOf fibo:pays`, so B's "total paid by customer" query traverses A's Actions transparently. Cross-org reconciliation becomes a graph query, not an ETL project.
**KR/standards mechanism.** Shared canonical predicate grounding as the interop substrate — same predicate IRI ⇒ zero-config interop.
**DEMAND.** The grounding IRI (not the local Verb name) must be the join key for cross-graph queries, and the interface must let two differently-named local Verbs (`settle`, `remit`) both declare `groundedAs: 'fibo:pays'`. Interop is keyed on `groundedAs`/`canonicalOf`, never on `name`.

### 10. Content-derived IDs giving the same Thing the same global identity across orgs
**Use case.** Org A and org B both have "Acme Corp, NAICS 541511, Delaware C-corp" in their graphs. Both mint the Thing via `deriveContentId('company', canonicalForm)` → identical `company_8f3a91b2c0e5`. When their graphs are fused into the dedup commons, the two records *collapse to one node* with no entity-resolution heuristics — the content hash already agreed. The NAICS code `541511` (Custom Computer Programming Services) is the `$type` discriminator that makes the canonical form deterministic.
**KR/standards mechanism.** Content-derived IDs (sha256-prefix-12 over canonical JSON) as **stable global identity** — the same canonical Thing gets the same address everywhere, independently minted; NAICS as the industry-class anchor.
**DEMAND.** Content-derived IDs must be deterministic across orgs/processes/machines (graphdl's `deriveContentId` already guarantees this), and the `Ontology()`/Provider `create()` path must support **content-addressed Thing IDs** (derive the id from a canonical projection of the data) so independent mints of the same canonical entity collide *intentionally*. The interface needs a documented "canonical projection" (which fields, normalized how) that feeds the hash — likely the `frozen`-stratum fields, since those are the identity-bearing ones.

### 11. Merging two graphs that independently minted the same predicate
**Use case.** Org A minted `verbs:onboards`; org B minted `verbs:onboard`. Both ground to the same canonical `verbs.org.ai/onboard` IRI. On graph fusion, the SVO layer recognizes them as the same predicate (same `canonicalOf`) and merges their Action streams under one logical edge type, while preserving each org's local label for display. The `manner` enum values from both (`{ self-serve, white-glove }` ∪ `{ guided, automated }`) union into the canonical predicate's known manner set.
**KR/standards mechanism.** Predicate-level reconciliation via shared canonical IRI; `owl:equivalentProperty` for the local→local link; SKOS-style label preservation.
**DEMAND.** Merging must reconcile on the canonical IRI while preserving per-org local Verb identity (so display labels survive), and `Frame.manner` enums must be **unionable** across orgs that share a canonical predicate — i.e. `manner` is an open extensible set governed at the canonical level, not a closed per-org list.

### 12. Versioning and migrating a Verb's Frame across ontology releases
**Use case.** `verbs.org.ai` ships `ships v1` with `Frame { subject: 'Company', object: 'Product' }`. In `v2`, it adds a `destination: 'Address'` role and deprecates the old `to` data field in favor of the structured `destination`. A startup pinned to `verbs.org.ai@1.x` keeps the old Frame; on `pnpm update` to `2.x`, a migration rewrites existing `ships` Actions to lift `data.to` into `roles.destination`. The ontology carries `owl:versionInfo` / `owl:deprecated` so consumers know what changed.
**KR/standards mechanism.** Semantic versioning of an ontology (`owl:versionIRI`, `owl:priorVersion`, `owl:deprecated`) + Frame-evolution migrations; canonical-registry version pinning.
**DEMAND.** A Verb/Frame must carry a **version stamp** and the canonical registry must be pin-able by semver (`verbs.org.ai@^2`). The interface needs a Frame-migration path (old roles → new roles) so evolving a canonical predicate's valency does not silently break Actions minted under the prior Frame. Deprecation must be expressible (`deprecated?: boolean` + `supersededBy?: IRI`) rather than destructive.

## Highest-leverage use case + sharpest DEMAND

**Highest leverage: #8 — losslessly exporting an `Ontology()` as a JSON-LD `@context` + `@graph` (with #9 as the payoff it unlocks).**

I hold this position firmly. Every other use case in this lens is *downstream of serialization*. Grounding (1, 2, 4, 5), inverses (7), domain/range (6), cross-org interop (9), and version migration (12) all presuppose that the vocabulary can leave the process boundary as standards-compliant linked data and come back without loss. If the `Ontology()` object only lives as an in-memory TypeScript structure, the million startups are a million private islands and the "shared canonical vocabulary" thesis evaporates. The moment the ontology serializes to a dereferenceable JSON-LD `@graph`, **the whole corpus becomes one queryable web**: a founder publishes their ontology at a URL, a second founder imports it, a reasoner materializes inverses and subsumptions, and two startups that independently grounded `pays` to `fibo:pays` reconcile their books with a graph traversal instead of an integration project.

**The 1000× moment:** every generated startup ships an `ontology.jsonld` the way today's sites ship a `sitemap.xml`. Crawl the million, fuse on canonical IRIs + content-derived IDs, and you have an **AI-native economy that transacts through one shared, grounded predicate vocabulary** — where any agent can discover what any startup *does* (its Verbs), *to what* (its Frames' ranges), and *for whom* (recipients), and compose them, because they all speak grounded SVO. That is the difference between a million apps and one economy.

**Sharpest DEMAND:** *The `Ontology()` object must serialize bijectively to/from a JSON-LD `@context` + `@graph`, with `Verb.source`/`canonical`/`groundedAs`, `Frame` roles (mapped to schema.org `Action` argument-roles), `Verb.inverse` (→ `owl:inverseOf`), Noun `$type`/`subClassOf`, and version/deprecation metadata ALL surviving the round-trip.* If one field drops, the artifact stops being trustworthy linked data and the interop commons degrades to lossy ETL.

## Requirements this lens imposes on the committed interface

- **Resolvable grounding IRI per Verb.** Add `groundedAs?: IRI` (and/or `canonicalOf?: IRI`) so a Verb anchors to an external/canonical registry entry (`onet:...`, `apqc:10015`, `fibo:pays`, `verbs.org.ai/...`) — `source` (the enum) alone is insufficient; interop and merge must key on the IRI, never on `Verb.name`.
- **Stable opaque grounding identity, decoupled from label.** The external anchor must be the immutable code (APQC's 5-digit element ID, O*NET task number), with `name`/conjugations as the mutable human layer — so upstream label renames don't break grounding.
- **`source` + `canonical` must be mutable post-mint, with a first-class promotion transition.** `domain/false → verbs.org.ai/true` must preserve local Verb identity and not orphan existing Actions referencing it by name.
- **Nouns need a `$type` IRI on the runtime interface.** The committed `Noun` lacks `$type`; it must gain one (inherited from graphdl), defaulting to `schema.org.ai/{Name}`, so a `Thing` serializes with `"@type"`.
- **Class/predicate subsumption slots.** `Noun.subClassOf?: IRI` and `Verb.subPropertyOf?: IRI` so local vocab can specialize imported standards (FIBO, schema.org) and fold under canonical parents in subsumption queries.
- **Frame roles must map to a published RDF argument-role vocabulary.** Align `Frame` role names to schema.org `Action` (`subject→agent`, `object→object`, `recipient→recipient`, `instrument→instrument`, plus `result`), so an exported Action *is* a valid `schema:Action`; `Frame.subject`/`object` export as `rdfs:domain`/`rdfs:range`.
- **`Verb.inverse` → `owl:inverseOf`; nested `reverse` carried to runtime.** Keep the structured `reverse: { by, at, in, for }` (per the DECISION) and surface it on the runtime Verb for inverse traversal + reverse-direction labels.
- **Lossless JSON-LD `@context` + `@graph` export/import on `Ontology()`.** A bijective serializer is the keystone requirement — every grounding, Frame role, `source`/`canonical`/`groundedAs`, `$type`, inverse, and version field must round-trip.
- **Content-addressed Thing IDs with a documented canonical projection.** Wire `deriveContentId` into the `create()` path so independent mints of the same canonical entity collide intentionally; define which fields (likely the `frozen` stratum) form the canonical projection feeding the hash.
- **Predicate/entity merge keyed on canonical IRI, preserving local labels.** Graph fusion reconciles on `canonicalOf`/`groundedAs` (with `owl:equivalentProperty` for local↔local), keeping each org's display label; `Frame.manner` is an open, unionable set governed at the canonical level.
- **Ontology versioning + deprecation as first-class.** Carry `owl:versionInfo`/`versionIRI`, `deprecated?: boolean`, `supersededBy?: IRI`; make the canonical registry pin-able by semver (`verbs.org.ai@^2`) with a Frame-evolution migration path (old roles → new roles).
